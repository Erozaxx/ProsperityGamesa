Prosperity Rebuild — High Level Design

Architektura projektu · věrný rebuild hry „Prosperity" jako mobile-first PWA, offline
iter-002 / T-001 (rev. T-003) Review: GO s úpravami · 0 blockerů Headless sim + preact UI ES2022 · runtime zero-build offline-first · IndexedDB

Shrnutí. Headless kroková simulace v čistých ES modulech (JS + JSDoc typy, bez build kroku) s tenkou preact UI vrstvou, local-first persistencí v IndexedDB a jediným fixed-timestep mechanismem pro live běh, background i offline catch-up. Přímá realizace konsolidovaného seznamu K0–K19 z analýzy iter-01: bází je jediný serializovatelný stav vlastněný simulací (K0), nad ním deklarativní scheduler a tick-orchestrace (K3/K6), data-driven katalogy s vrstvou modifikátorů (K2/K13–K15), balanc jako data + čisté vzorce (K4), registry transakce (K5) a fail-fast string-ID registr (K10).

Krok = 0,05 s · 900 kroků/den · sezóna 91 dní. Rychlosti Pause/1×/2×. Interaktivní eventy umí dávku přerušit (stopPending).

tickOrder uvnitř jednoho kroku (rozpad Home.step — živý artefakt)

Periodika registrovaná jako data {id, every, order, systemFn}; engine počítá hrany jednou za krok → žádné modulo prology v hot-path. RNG: pojmenované streamy per systém, stav v save (reprodukovatelný catch-up).

Persist = deklarativní allowlist schéma per doména (jen dynamika), verzované migrace, load = čistá konstrukce. Autosave: perioda + pagehide/visibilitychange + po událostech. 5 · Registr klíčových rozhodnutí (D1–D13)

§2D1Stack: ES2022 + JSDoc + tsc --checkJs, runtime zero-build; vendor preact+htm; node:test; ruční SW.
§3D2Vrstvení: headless core ↔ UI přes read-only snapshot + command API.
§4D3Engine: fixed-timestep 0,05 s + akumulátor; min-heap schedule + periodika; centrální tickOrder.
§4.4D4RNG: seedovatelný PRNG, pojmenované streamy per systém, stav v save.
§5D5Katalogy: verzované JSON, schema validace, immutable + modifikátory, efekty jako string-ID.
§6D6Save: IndexedDB, rotující generace, lastSimTimestamp, allowlist schéma, migrace, export string.
§7D7Transakce: registry resourceHandlers[kind]; účetnictví a achievementy jako observery.
§8D8Bitva: deterministický serializovatelný automat (sub-step 30 ms), auto-resolve v catch-upu.
§9.1D9R1: klientský trh — lokální available + denní mean-reversion drift; vzorec ceny 1:1.
§9.2D10R2: cap catch-upu = technický strop 8 h (benchmark M0) + oddělená balanční hodnota (M9).
§9.3D11R3: extrakční pipeline = M1 s gap-reportem; eskalace jen pro nedotěžitelné díry.
§9.4D12R4: kontrakty pozdních systémů teď + stub registrace + kontraktní testy od M2; impl. M7.
§4.6D13Simulace v main threadu (frame budget); Web Worker jen jako eskalační cesta po benchmarku.
